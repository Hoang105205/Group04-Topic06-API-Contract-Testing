Why API Testing?

02 / 16
01

System Backbone

APIs connect Frontend, Backend, and third-party integrations. An API failure paralyzes the entire system immediately.

02

Shift-Left Testing

Detect and isolate bugs early in the service logic layer. Fixing API bugs is far cheaper than fixing them at the GUI level.

03

Speed & Automation

Execute hundreds of test cases in seconds. Easily integrate into CI/CD pipelines for automated regression testing.

CORE REALIZATION: A critical API bug on the Backend can crash all client platforms (Web, iOS, Android) simultaneously before the UI can handle or display any structured error message.

API Authentication Testing Patterns

07 / 16

Common Auth Mechanisms

  • Bearer Token (JWT): Token generated by the server and sent in the Authorization Header. Standard for SPA and Mobile apps.
  • API Key: Static key passed via Header/Query.
  • Basic Auth: Base64 encoded credentials sent directly.
  • Session Cookie: Server maintains session, browser sends cookie automatically.

EShop SUT Automation Workflow

  • Step 1: POST /api/login sends credentials to server.
  • Step 2: Post-response script extracts token and auto-saves it in {{auth_token}} environment variable.
  • Step 3: All protected child requests inherit from the Collection level automatically (Auth Inheritance).

Scripting (Pre-request vs. Post-response)

08 / 16

Pre-request Scripts (Before sending)

  • Purpose: Prepare dynamic data or calculate inputs before the request is sent.
  • Real-world Example: Generating a random email during `POST /api/register` to avoid 'Email already exists' errors:
const randomEmail = `test_${Math.floor(Math.random() * 10000)}@eshop.com`;
pm.environment.set("email", randomEmail);

Then use `{{email}}` directly in the Request Body JSON.

Post-response Scripts / Tests (After receiving)

  • Purpose: Validate API response quality (assertions) and extract variables for workflow chaining.
  • Variable Chaining: Extract login token and user ID to pass them to GET Cart, POST Cart, and Checkout endpoints.
const responseData = pm.response.json();
pm.test("Save Auth Token & User ID", function () {
    pm.environment.set("auth_token", responseData.token);
    pm.environment.set("user_id", responseData.user.id);
});

* This will be demonstrated live on Postman during the demo phase.

Contract Testing with Postman (API Interface Validation)

09 / 16

JSON Schema Validation

Postman integrates the Ajv Validator (v6.12.5) to support JSON Schema draft-07.

Asserts key data types (integer, string), required fields, and format constraints (format: email).

Structure Assertions

Validates other aspects of the API contract:

  • Status Code: Checks if the correct HTTP status is returned.
  • Headers: Verifies the response format (e.g., application/json).
  • SLA Response: Ensures the API meets performance limits (<800ms).

Living Contracts

Save sample responses as Examples directly in Postman.

These Examples serve as the API documentation, which can spin up Postman Mock Servers for Frontend/Backend decoupled development.

Postbot AI in Automated Test Generation

11 / 16

Interaction Flow

  • 1. Send request to get sample response.
  • 2. Go to Tests tab and click 'Ask Postbot'.
  • 3. Type natural language prompt.
  • 4. Review and save assertions.

Custom Prompting Power

To test complex calculations instead of basic types, write specific calculations:

"Write a test that calculates expected discount as total_amount * 10 / 100 for a percent coupon, and asserts discount_amount matches this calculation."

With custom guidance, Postbot writes logical code instead of simple type checks.

Limitations of Postbot AI

12 / 16
LIMIT 1

Happy-Path Focus

Postbot is heavily biased towards generating successful (status 200/201) tests. It completely misses negative error paths and edge cases where business logic bugs usually reside.

LIMIT 2

Sample Dependency

AI works by analyzing the active sample response. If the request has not been run, or if the server is down, Postbot cannot generate assertions.

LIMIT 3

No Business Logic

By default, AI only checks surface-level structure (strings, numbers). It cannot verify numerical correctness or stateful business rules without custom prompts.

Failure Modes — 3 Ways Tools Mislead You

13 / 16
FM 1

Bug Anchoring

Postbot 'learns' and locks in server bugs instead of catching them. If the sample response has wrong calculations (e.g., negative discount -4,500,000₫), Postbot writes tests validating this bug as correct behavior.

FM 2

Superficial Error Testing

AI-generated error tests only check if the error property is a string, completely ignoring whether the error message matches the actual business context or stateful limits.

FM 3

Environment Mismatch

Collection Runner's environment selector is separate from the main workspace. Mismatched active environments cause variable resolution failures, which the Runner hides.
